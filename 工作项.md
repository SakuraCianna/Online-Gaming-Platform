  - [ ] RabbitMQ死信队列实现60秒延迟

- **为什么2秒匹配一次**：平衡响应速度和服务器压力

### Week 7-8：Redis缓存优化

#### 任务列表
- [ ] **缓存穿透防护**
  - [ ] 优化现有布隆过滤器（添加到UserService、GameRoomService）
  - [ ] 对不存在的数据缓存空值（TTL 60秒）
  - [ ] 编写测试：查询10000个不存在的用户，数据库访问次数 = 0
  
- [ ] **缓存击穿防护**
  - [ ] 热点数据（用户信息、房间信息）使用分布式锁
  - [ ] 使用Redisson实现可重入锁
  - [ ] 双重检查：加锁前后都检查缓存
  
- [ ] **缓存雪崩防护**
  - [ ] 所有缓存TTL加随机值（±10%）
  - [ ] 热点数据永不过期 + 后台定时刷新
  - [ ] Redis哨兵模式（可选，本地开发可跳过）
  
- [ ] **缓存一致性**
  - [ ] 数据更新时先删除缓存
  - [ ] 延迟双删（删除 -> 更新DB -> 等待1秒 -> 再删除）
  - [ ] Canal监听MySQL binlog自动刷新缓存（可选）

#### 验收标准
- [ ] 用JMeter压测1000并发查询用户信息，缓存命中率 > 95%
- [ ] 热点数据过期时，只有1个请求访问数据库
- [ ] 10000个缓存同时过期，数据库QPS < 100

#### 技术亮点
- **三大缓存问题的区别**：穿透（数据不存在）、击穿（热点过期）、雪崩（大量过期）
- **为什么用Redisson**：支持可重入锁、看门狗机制（自动续期）
- **延迟双删的原理**：解决并发读写导致的缓存不一致

  - [ ] 四个维度：日榜、周榜、月榜、总榜

- [ ] **排行榜查询接口**
  - [ ] GET `/api/leaderboard/{game}/{period}` - 获取Top100
  - [ ] GET `/api/leaderboard/{game}/{period}/rank/{userId}` - 查询个人排名
  - [ ] GET `/api/leaderboard/{game}/{period}/around/{userId}` - 查询前后5名
  
- [ ] **前端展示**
  - [ ] 排行榜页面（卡片式布局）
  - [ ] 实时更新（WebSocket推送排名变化）
  - [ ] 高亮当前用户
  - [ ] 支持切换游戏和时间维度

#### 验收标准
- [ ] 查询Top100，响应时间 < 50ms
- [ ] 查询个人排名，响应时间 < 10ms
- [ ] 1000人同时游戏结束，排行榜更新无延迟

#### 技术亮点
- **为什么用ZSet**：自动排序，ZREVRANGE查询Top N复杂度O(log(N)+N)
- **为什么只保留Top100**：节省内存，大部分玩家只关心前排
- **个人排名查询优化**：ZREVRANK时间复杂度O(log(N))，比MySQL快

### Week 11-12：RabbitMQ异步处理

#### 任务列表
- [ ] **游戏结算异步化**
  - [ ] 游戏结束不直接写数据库，发送到MQ
  - [ ] 消费者处理：保存记录 -> 更新积分 -> 更新排行榜 -> 检查成就
  - [ ] 单次游戏处理时间从200ms降到50ms
  
- [ ] **消息可靠性保证**
  - [ ] 生产者确认（ConfirmCallback）

  - [ ] 死信队列消息记录到数据库，人工处理
  
- [ ] **邮件发送异步化**
  - [ ] 注册验证码、密码重置邮件通过MQ发送
  - [ ] 失败自动重试，记录发送日志
  - [ ] QQ邮箱限流（每秒最多1封）
  
- [ ] **好友通知优化**
  - [ ] 好友请求、消息通过MQ + WebSocket推送
  - [ ] 离线消息存储，上线后批量推送
  - [ ] 消息去重（布隆过滤器）

#### 验收标准
- [ ] 100场游戏同时结束，所有结算在5秒内完成
- [ ] 消息丢失率 = 0（通过手动ACK保证）
- [ ] 邮件发送失败能自动重试

#### 技术亮点
- **为什么异步处理**：削峰填谷，避免数据库压力过大
- **手动ACK的作用**：消费失败时消息重新入队，保证可靠性
- **死信队列的用途**：处理反复失败的消息，避免阻塞正常流程

## 🎯 项目完成后的简历描述模板

### 项目标题
```
在线游戏平台 | Spring Boot 3.5 + Vue 3 + Redis + RabbitMQ
个人独立开发 | 2024.08 - 2025.02 | 代码量约2万行
```

### 项目描述
```
一个支持多人实时对战的Web游戏平台，包含2048、五子棋、扫雷、坦克大战4款游戏，
实现了用户系统、好友系统、房间匹配、实时排行榜等功能。
```

### 核心技术亮点
```
1. 【实时对战】使用WebSocket实现坦克大战60FPS游戏状态同步，服务端权威碰撞检测防作弊，
   支持4人同时对战，平均延迟<100ms

2. 【智能匹配】基于Redis Sorted Set实现ELO积分匹配系统，2秒自动配对积分相近玩家，
   RabbitMQ死信队列处理60秒匹配超时

3. 【实时排行榜】使用Redis ZSet实现日/周/月/总四维度排行榜，支持Top100查询（<50ms）
   和个人排名查询（<10ms），QPS达到1000+

4. 【缓存优化】实现布隆过滤器防穿透、Redisson分布式锁防击穿、随机TTL防雪崩，
   缓存命中率95%以上，减少90%数据库访问

5. 【异步处理】RabbitMQ异步处理游戏结算、邮件发送，削峰填谷，单次请求耗时从200ms降至50ms，
   手动ACK保证消息零丢失

6. 【性能优化】JMeter压测登录接口QPS达到500+，数据库索引优化后查询速度提升10倍，
   单元测试覆盖率52%
```

### 技术栈
```
后端: Spring Boot 3.5、Java 21、MyBatis-Plus、Spring Security、JWT
数据库: PostgreSQL 18、Redis 7
消息队列: RabbitMQ 3
实时通信: WebSocket (STOMP)
前端: Vue 3、Element Plus、Phaser 3
部署: Docker Compose、Nginx
测试: JUnit 5、Mockito、JMeter
```

## ⚠️ 风险与应对

### 可能遇到的问题
1. **时间不够**：优先完成核心功能（坦克大战+排行榜），其他可选
2. **技术难度过高**：从简单场景开始，逐步优化（先用SimpleBroker，再升级RabbitMQ）
3. **测试覆盖率低**：至少覆盖核心Service的关键方法
4. **性能达不到目标**：记录实际数据，分析瓶颈即可

### 最小可行方案（MVP）
如果时间紧张，必须完成：
- ✅ 坦克大战基本功能（不要求60FPS，30FPS也行）
- ✅ 排行榜系统（至少实现周榜）
- ✅ 5个核心Service的单元测试
- ✅ 简单的性能测试报告
