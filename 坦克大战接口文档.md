# 坦克大战后端接口需求文档

## 一、REST API 接口

### 1.1 TankBattle.vue（房间准备页）需要的接口

| 接口路径                           | 请求方式 | 参数            | 参数说明                          | 返回数据                                                  | 作用                                                         |
| ---------------------------------- | -------- | --------------- | --------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------ |
| `/api/tankbattle/{roomCode}/start` | POST     | `userId` (Long) | 房主用户ID                        | `{ success, code, message, data: { gameId, startTime } }` | 房主启动游戏，初始化游戏状态到Redis，启动服务端60FPS游戏循环 |
|                                    |          | `mode` (String) | 游戏模式（1v1v1v1/2v2）           |                                                           |                                                              |
|                                    |          | `map` (String)  | 地图名称（classic/desert/forest） |                                                           |                                                              |

### 1.2 通用房间接口（复用现有的 GameRoomController）

| 接口路径           | 请求方式 | 参数                                         | 参数说明         | 返回数据                                         | 作用                 |
| ------------------ | -------- | -------------------------------------------- | ---------------- | ------------------------------------------------ | -------------------- |
| `/api/room/invite` | POST     | `userId`, `friendId`, `roomCode`, `gameName` | 邀请好友参数     | `{ success, code, message }`                     | 邀请好友加入房间     |
| `/api/room/addAI`  | POST     | `roomCode`, `seatIndex`                      | 房间码、座位索引 | `{ success, code, message, data: { aiPlayer } }` | 向指定座位添加AI玩家 |

## 二、WebSocket 接口

### 2.1 TankBattle.vue（房间准备页）需要订阅的频道

| 频道路径                                 | 通信方向      | 消息格式                                                                                      | 作用                                         |
| ---------------------------------------- | ------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------- |
| `/topic/tankbattle/{roomCode}/gameStart` | 服务端→客户端 | `{ gameId, mode, map, players: [{ userId, username, playerId, spawnX, spawnY }], startTime }` | 游戏启动成功通知，客户端收到后跳转到游戏页面 |

### 2.2 TankBattleGame.vue（游戏页）需要的 WebSocket 频道

| 频道路径                                  | 通信方向      | 消息格式                                                                                       | 发送/接收频率      | 作用                                             |
| ----------------------------------------- | ------------- | ---------------------------------------------------------------------------------------------- | ------------------ | ------------------------------------------------ |
| `/app/tankbattle/{roomCode}/playerAction` | 客户端→服务端 | `{ userId, keys: { w, s, a, d, k }, timestamp }`                                               | 60次/秒            | 发送玩家键盘操作到服务端                         |
| `/topic/tankbattle/{roomCode}/gameState`  | 服务端→客户端 | `{ frame, timestamp, tanks: [{ id, x, y, r, h, d }], bullets: [{ id, x, y, vx, vy }] }`        | 60次/秒（16.67ms） | 接收服务端计算的游戏状态并渲染                   |
| `/topic/tankbattle/{roomCode}/gameEvent`  | 服务端→客户端 | `{ type, ...eventData }`                                                                       | 事件触发时         | 接收游戏事件（击杀kill、死亡death、重生respawn） |
| `/topic/tankbattle/{roomCode}/gameEnd`    | 服务端→客户端 | `{ gameId, reason, duration, finalStats: [{ userId, username, kills, deaths, score, rank }] }` | 游戏结束时         | 接收游戏结算数据，显示结算界面                   |

## 三、数据结构详细说明

### 3.1 playerAction 消息格式

```json
{
  "userId": 123,
  "keys": {
    "w": true,    // 前进
    "s": false,   // 后退
    "a": false,   // 左转
    "d": false,   // 右转
    "k": false    // 射击
  },
  "timestamp": 1698765432000
}
```

### 3.2 gameState 消息格式（精简版，减少带宽）

```json
{
  "frame": 12345,
  "timestamp": 1698765432000,
  "tanks": [
    {
      "id": 0,        // 玩家索引（0-3）
      "x": 150.5,     // 位置x
      "y": 200.3,     // 位置y
      "r": 45.2,      // 旋转角度
      "h": 200,       // 血量
      "d": 0          // 是否死亡（0=存活，1=死亡）
    }
  ],
  "bullets": [
    {
      "id": "b1",
      "x": 300,
      "y": 400,
      "vx": 10,
      "vy": 5
    }
  ]
}
```

### 3.3 gameEvent 消息格式

#### 击杀事件
```json
{
  "type": "kill",
  "killerId": 123,
  "victimId": 456,
  "weapon": "bullet",
  "timestamp": 1698765432000
}
```

#### 死亡事件
```json
{
  "type": "death",
  "playerId": 456,
  "killerId": 123,
  "respawnTime": 3
}
```

#### 重生事件
```json
{
  "type": "respawn",
  "playerId": 456,
  "x": 150,
  "y": 150,
  "health": 300
}
```

## 四、Redis 数据结构设计

| Key                                         | 类型   | 字段/内容                                                   | TTL                                  | 作用                         |
| ------------------------------------------- | ------ | ----------------------------------------------------------- | ------------------------------------ | ---------------------------- |
| `tankbattle:room:{roomCode}:config`         | Hash   | `mode`, `map`, `maxPlayers`, `respawnTime`                  | 24小时                               | 存储房间游戏配置             |
| `tankbattle:game:{roomCode}:state`          | Hash   | `status`, `startTime`, `frame`, `tanks`, `bullets`, `walls` | 结束后5分钟                          | 存储游戏运行时状态           |
| `tankbattle:game:{roomCode}:input:{userId}` | String | `{ keys: {...}, timestamp }`                                | 1秒                                  | 存储玩家最新操作（高频更新） |
| `tankbattle:leaderboard:{period}`           | ZSet   | score=总积分, member=userId                                 | 日榜1天，周榜7天，月榜30天，总榜永久 | 存储排行榜数据               |

## 五、RabbitMQ 消息队列设计

| 队列名称                        | 消息格式                                                                                     | 消费者处理逻辑                                                                                     | 作用                                   |
| ------------------------------- | -------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- | -------------------------------------- |
| `tankbattle.game.settlement`    | `{ gameId, roomCode, duration, players: [{ userId, kills, deaths, score, rank }], endTime }` | 1. 批量插入数据库（tank_battle表）<br>2. 更新用户统计<br>3. 发送消息到排行榜更新队列<br>4. 手动ACK | 异步处理游戏结算，避免阻塞游戏结束流程 |
| `tankbattle.leaderboard.update` | `{ userId, score, kills, deaths, timestamp }`                                                | 1. 更新4个排行榜（日/周/月/总）<br>2. 清理过期榜单<br>3. 手动ACK                                   | 异步更新排行榜，削峰填谷               |

## 六、后端核心实现要点

### 6.1 游戏循环（服务端权威）

- **定时任务**：`@Scheduled(fixedRate = 16)`，每16ms执行一次（约60FPS）
- **处理流程**：
  1. 从Redis加载游戏状态
  2. 获取所有玩家的最新输入
  3. 更新游戏逻辑（坦克移动、碰撞检测、子弹飞行、AI行为）
  4. 保存到Redis
  5. 通过WebSocket广播到所有客户端

### 6.2 AI逻辑（服务端实现）

- **移动**：5%概率随机改变方向
- **射击**：2%概率随机射击
- **无智能**：不进行目标追踪、躲避等高级AI行为

### 6.3 消息限流（防止客户端攻击）

- 使用Redis记录每个用户的消息发送次数
- 限制：每秒最多100次操作
- 超过限制抛出429异常

## 八、技术亮点总结

| 技术点                | 实现方式                     | 面试回答要点                                                  |
| --------------------- | ---------------------------- | ------------------------------------------------------------- |
| **WebSocket实时通信** | STOMP协议，60FPS状态同步     | 相比HTTP轮询，减少无效请求，实时性更强，支持双向通信          |
| **服务端权威架构**    | 所有游戏逻辑在服务端计算     | 防止客户端修改代码作弊，保证游戏公平性                        |
| **Redis缓存优化**     | 游戏状态存Hash，排行榜用ZSet | Hash避免频繁序列化，ZSet自动排序，查询Top100复杂度O(log(N)+N) |
| **RabbitMQ异步处理**  | 游戏结算、排行榜更新         | 削峰填谷，避免数据库压力过大，请求耗时从200ms降至50ms         |
| **消息限流**          | Redis计数器，每秒最多100次   | 防止恶意客户端攻击，保护服务器资源                            |
| **性能优化**          | 精简JSON字段名，只发变化数据 | 减少带宽占用，提升传输效率                                    |
